================================================================================
  1. PROJECT OVERVIEW
================================================================================

CampusArena is a web application for college students and faculty to:

  • Browse and discover all college events (hackathons, workshops, coding
    contests, design jams, quizzes, paper presentations, etc.)

  • Register for events — individually or as a team.

  • Form teams with smart tech-stack matching — when registering for team
    events, users specify their tech skills (Frontend, Backend, ML/AI,
    DevOps, UI/UX, Mobile, etc.), ensuring diverse skill sets in teams.

  • Create teams & manage join requests — team creators can approve or
    decline incoming join requests from other students.

  • Real-time updates — see registered participants, team openings,
    and live stats update in real time (no page refresh needed).

  • In-team messaging — team members can chat in real time to coordinate
    and collaborate before, during, and after events.

  • Event details — each event shows: description, rules, timeline,
    prizes, eligibility, participation certificate availability, judges,
    sponsors, FAQs, and more.

─── MODULE 4: REGISTRATION SYSTEM ───

  [F4.1] Solo Registration
         - User clicks "Register Now" on event
         - Fills the custom registration form defined by organizer
         - Confirms registration
         - Receives confirmation email
         - Appears in event's participant list in real time

  [F4.2] Team Registration
         - Two flows:

         FLOW A: "Create a Team"
           1. User enters Team Name
           2. Fills registration form (personal fields)
           3. Selects their tech stack / role in team
           4. Team is created — user becomes TEAM LEADER
           5. Team appears in "Teams with Open Spots" (if spots remain)
           6. Other users can send "Join Request" to this team
           7. Team Leader sees incoming requests with:
              • Requester's name, profile, tech stack, skills
              • "Approve" / "Decline" buttons
           8. When team is full (max members reached), it auto-closes
              for new requests

         FLOW B: "Join an Existing Team"
           1. User browses "Teams with Open Spots" in the event
           2. Each team card shows:
              • Team Name
              • Current Members (count & names)
              • Tech Stacks already in team
              • Tech Stacks needed (calculated: max_size - current_size)
              • Team Leader name
           3. User clicks "Request to Join"
           4. Fills registration form (personal fields)
           5. Selects their tech stack / role
           6. Request sent to Team Leader for approval
           7. If approved → user added to team, appears in members list
           8. If declined → user notified, can try other teams

         FLOW C: "Register as Individual, Find Team Later"
           1. User registers individually with "Looking for Team" flag
           2. Appears in "Looking for Team" section on event page
           3. Team leaders can invite these users
           4. User can send join requests to teams

  [F4.3] Tech Stack Selection (Critical for Team Events)
         - During registration for team events, each member MUST select:
           • Primary Tech Stack (1-3 technologies they are best at)
           • Secondary Tech Stack (additional skills)
           • Preferred Role:
             □ Frontend Developer
             □ Backend Developer
             □ Full Stack Developer
             □ Mobile Developer
             □ UI/UX Designer
             □ ML/AI Engineer
             □ Data Scientist
             □ DevOps Engineer
             □ Project Manager
             □ Other: ________
         - This ensures teams have DIVERSE skill sets
         - Team view shows tech stack distribution:
           e.g., "Team Alpha: React ✓ | Node.js ✓ | ML ✗ | Design ✗"
         - Helps build a real, balanced product team

  [F4.4] Registration Confirmation
         - On-screen confirmation with animation
         - Email confirmation with event details
         - Add to Calendar option (.ics file download)
         - Unique Registration ID generated

================================================================================
  6. REAL-TIME FEATURES (WebSocket Architecture)
================================================================================

Technology: Django Channels (ASGI) + WebSockets
Channel Layer: Redis (via channels_redis)
Server: Daphne (ASGI server for Django Channels)

Django Channels uses "Consumers" (similar to Django views but for
WebSocket connections) and "Groups" (for broadcasting to multiple
connections). Each consumer handles connect, disconnect, and receive
events.

─── REAL-TIME EVENT 1: REGISTRATION UPDATES ───
  Group: `event_{eventId}_registrations`
  Consumer: RegistrationConsumer (channels routing)
  Triggered when: A new user registers for an event
  Payload: {
    registrationCount: 45,
    newParticipant: { name, college, type: 'individual'|'team' },
    teamUpdate: { teamId, teamName, memberCount } (if team reg)
  }
  Listeners: Event detail page (Tab 4: Registered Participants)

─── REAL-TIME EVENT 2: TEAM UPDATES ───
  Group: `event_{eventId}_teams`
  Consumer: TeamConsumer
  Triggered when: Team created, member joins/leaves, team opens/closes
  Payload: {
    action: 'team_created' | 'member_joined' | 'member_left' |
            'team_closed' | 'team_opened',
    team: { id, name, memberCount, maxMembers, isOpen, techStacks }
  }

─── REAL-TIME EVENT 3: JOIN REQUESTS ───
  Group: `team_{teamId}_requests`
  Consumer: JoinRequestConsumer
  Triggered when: Join request sent, approved, or declined
  Payload: {
    action: 'new_request' | 'approved' | 'declined',
    request: { userId, userName, techStacks, role, message }
  }
  Listeners: Team leader's dashboard

─── REAL-TIME EVENT 4: TEAM CHAT ───
  Group: `team_{teamId}_chat`
  Consumer: ChatConsumer
  Triggered when: Team member sends a message
  Payload: {
    messageId, senderId, senderName, senderAvatar,
    content, timestamp, messageType
  }
  Additional groups:
    `team_{teamId}_typing` — typing indicator
    `team_{teamId}_read`   — read receipts

─── REAL-TIME EVENT 5: NOTIFICATIONS ───
  Group: `user_{userId}_notifications`
  Consumer: NotificationConsumer
  Triggered when: Any notification triggers
  Payload: { notification object }

  9. SECURITY CONSIDERATIONS
================================================================================

  [S1] Password Hashing: Django's PBKDF2 hasher (default); can add
       bcrypt via Django's PASSWORD_HASHERS setting
  [S2] Input Validation: Django Forms + DRF serializers for server-side;
       HTML5 + JS for client-side validation
  [S3] SQL Injection: Django ORM (parameterized queries by default)
  [S4] XSS Prevention: Django templates auto-escape by default;
       sanitize rich text content (e.g., bleach library)
  [S5] CSRF Protection: Django's built-in CsrfViewMiddleware +
       {% csrf_token %} template tag in all forms
  [S6] Rate Limiting: django-ratelimit or DRF throttling classes
  [S7] File Upload Security: Validate file types & sizes via Django
       FileField validators; scan for malware in production
  [S8] Session Security: Django sessions (server-side), secure &
       HTTP-only session cookies, SESSION_COOKIE_AGE configuration
  [S9] CORS: django-cors-headers package for allowed origins
  [S10] HTTPS: Enforce via SECURE_SSL_REDIRECT, SECURE_HSTS_SECONDS
       and Nginx configuration in production
  [S11] Role-Based Access Control: Django's built-in permissions +
       groups system, custom decorators / mixins for views
  [S12] Data Privacy: Careful serializer field selection in DRF;
       Django's @login_required and permission decorators
